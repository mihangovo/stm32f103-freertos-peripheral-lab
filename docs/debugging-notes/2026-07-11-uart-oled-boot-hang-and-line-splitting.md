# 调试复盘：OLED_Init 卡死 + 串口多行显示错乱

日期：2026-07-11
涉及提交：见 `git log` 中 `Core/Src/usart.c` / `Core/Src/freertos.c` / `User/TASK/ui_task.c` 相关的几次修改

## 问题一：合并 UART 实时监视功能后，`OLED_Init()` 里的 `HAL_Delay` 永久卡死

### 现象

- 单步调试发现程序卡在 `OLED_Init()` 内部 `HAL_Delay(200)` 的 `while ((HAL_GetTick() - tickstart) < wait) {}` 循环里，永远走不出来。
- 注释掉这一处 `HAL_Delay` 后，卡死点转移到了后面 `MPU_Init()` 里的另一处 `HAL_Delay`——说明不是某一次延时坏了，而是从某个时间点起，**所有**依赖 tick 的延时都会永久卡死。
- 这个问题在合并"UART 实时监视"功能的分支之前不存在。

### 排查过程

1. 先确认改动范围：完整比对分支 diff，确认 `HAL_Delay`、HAL 时基源配置（`stm32f1xx_hal_timebase_tim.c`，本工程用 **TIM8** 而不是 SysTick 做 HAL 的时基，这样可以把 SysTick 留给 FreeRTOS 自己用）、`main.c` 的初始化顺序等文件**完全没有被这次分支修改**。
2. 既然被卡住的代码本身没变，说明问题出在"新执行了什么代码"。逐行核对 `main()` 里到卡死点为止新增的调用，发现新分支在 `UART_Rx_Start()`（`main()` 里第111行左右调用，早于 `osKernelInitialize()`）里新增了一行：
   ```c
   UartLineQueueHandle = osMessageQueueNew(4, sizeof(UartLine_t), NULL);
   ```
   这是这次改动里**唯一**一个在 FreeRTOS 调度器真正启动（`osKernelStart()`）之前创建的新内核对象，其余队列/互斥锁全部是在 `MX_FREERTOS_Init()`（同样在 `osKernelStart()` 之前，但是这个位置在本工程里已经被证明是安全的）里创建的。
3. 没有在这台机器上继续深挖 FreeRTOS 移植层（`port.c`）里 `uxCriticalNesting` 临界区计数在调度器真正启动前的具体行为细节，而是采用了更稳妥的排查方式：**既然同样的 API 调用在 `MX_FREERTOS_Init()` 这个位置已经被证明是安全的（用了几个月都没问题），那就把新队列的创建挪到同样的位置**，而不是纠结于精确的底层机制。

### 根因（结论）

在 FreeRTOS 调度器真正启动（`osKernelStart()`/`vTaskStartScheduler()`）之前，某些 FreeRTOS 内核对象创建路径对"提前调用"这件事并不是在所有调用点都同样安全——具体来说，本工程里 `MX_FREERTOS_Init()`（`osKernelInitialize()` 之后、`osKernelStart()` 之前）这个位置是经过验证的安全区，而在更早的用户代码（`UART_Rx_Start()`，`main()` 里第111行）里调用同一个 API，会导致中断被永久屏蔽，进而让所有依赖 tick 中断的 `HAL_Delay` 全部卡死。

### 修复

- `Core/Src/usart.c`：`UART_Rx_Start()` 不再创建队列，只保留原有的"打开 DMA 空闲接收"逻辑。
- `Core/Src/freertos.c`：把 `UartLineQueueHandle = osMessageQueueNew(...)` 挪到 `MX_FREERTOS_Init()` 里已有的 `/* USER CODE BEGIN RTOS_QUEUES */ ... /* USER CODE END RTOS_QUEUES */` 区域，跟 `StorageCmdQueueHandle` 在同一个位置、同一个时机创建。

### 经验教训

- **FreeRTOS 内核对象的创建时机不是"只要在 `osKernelStart()` 之前就都一样安全"**——不同的调用点（哪怕都在调度器启动之前）可能有微妙的差异。最稳妥的做法是：新对象的创建**紧挨着已经验证过能正常工作的同类对象**放在一起，而不是图方便随手放在"看起来更早、更合理"的位置。
- 排查这类"从某个点开始，后面所有类似操作都失败"的问题时，要往"某个全局状态从此被破坏"（这里是中断被永久屏蔽）的方向想，而不是纠结于第一个复现现象本身（这里第一次复现是在 `OLED_Init`，但 `OLED_Init` 自己完全没有问题）。
- 定位思路：先确认"卡住的代码本身有没有被改过"（没有），再找"这次改动里唯一新执行的代码是什么"（只有一行），把怀疑范围锁定到最小。

## 问题二：串口一次收到多行数据时，OLED 显示错位、内容被截断

### 现象

- 连续快速发送多条消息（比如短时间内连发 `arch`、`three` 好几次），OLED 上"UART"实时监视页面里第一条消息出现在左下角而不是左上角，而且像 `three` 这样的内容显示成了 `ree`（丢字符）。
- Storage 历史记录页面翻到对应记录时，同样的内容也显示成 `ree`。

### 排查过程

1. 看串口打印出来的原始接收日志：`UART Rx Size=56`，内容是 `arch\r\narch\r\nthree\r\nthree\r\nthree\r\nthree\r\nthree\r\n1\r\n2\r\n3\r\n` 这样**一次性**收到的一大块，而不是一行一次回调。说明：如果发送间隔小于 UART 空闲判定的阈值，DMA 空闲事件回调 `HAL_UARTEx_RxEventCallback` 会把好几行数据打包成一次 `Size` 事件，行与行之间只是内嵌了 `\r\n`，并没有被自动拆开。
2. 但当时的代码（保存历史记录、推送到 UART 监视队列）是把整个 `Size` 字节的数据当成**一条**记录处理的。
3. 而 OLED 的 `OLED_ShowString` 遇到 `\r`/`\n`（ASCII 值小于空格）会直接停止绘制（`while((*chr>=' ')&&(*chr<='~'))`），不会跳过继续画后面的字符。
4. 换行/翻页逻辑 `OLED_ShowStringWrapped`/`UartMon_PushWrapped` 是按**固定字符数**切块前进的，并不知道中间还嵌着 `\r\n`。结果就是：显示会在某个内嵌的 `\r\n` 处戛然而止，但切割的读取指针已经按固定字符数跳过去了——下一段的起点不再是某一行真正的开头，而是切在了某个单词中间，于是 `three` 就被切出了 `ree` 这种烂尾片段。
5. "第一条消息显示在左下角"是另一个独立小问题：监视页面的滚动缓冲固定"往最后一行写新内容、其它行往上顶"，哪怕缓冲区还是空的也这么做，所以第一条消息永远先出现在最下面，而不是从左上角开始往下填。

### 根因（结论）

- **真正的行边界（`\r\n`）跟"一次 DMA 空闲事件收到的数据块"不是一回事**——一次事件可能包含零到多条完整的行。旧代码把"一次事件"误当成了"一行"。
- 显示层的自动换行逻辑只按固定字符数切分，没有对齐到真正的行边界，遇到内嵌的 `\r\n` 时会跟 `OLED_ShowString` 的"遇控制符即停"行为产生错位。

### 修复

- `Core/Src/usart.c`：`HAL_UARTEx_RxEventCallback` 里新增按 `\n`（连同前面的 `\r` 一起）拆分原始数据的逻辑，一次事件里有几条真正的行，就分别调用几次"保存历史记录 + 推送监视队列"，而不是整块转发。
- `User/TASK/ui_task.c`：监视页面的滚动缓冲增加一个"已填充行数"计数，未填满 4 行之前直接往下一行追加，填满后才开始整体上移滚动（而不是永远只往最后一行写）。

### 经验教训

- 处理任何"分包/分块"到来的流式数据时，不能假设"一次回调 = 一条逻辑记录"，必须显式按照约定的分隔符（这里是 `\r\n`）切分，尤其是在数据源可能因为发送节奏变化而合并多条记录到一次传输里的场景（这里是 UART 空闲检测触发的可变长度 DMA 传输）。
- 排查"内容被诡异截断/错位"类问题时，要同时看两层：**数据在哪被切的**（应用层的分块/换行逻辑）和**显示函数本身怎么处理特殊字符的**（`OLED_ShowString` 遇控制符即停），两者结合起来才能定位到真正的错位点，只看其中一层容易得出"字符宽度算错了"这类似是而非的结论。

## 问题三：DMA 循环模式下"重新武装"导致数据重复处理（比问题二更深一层的根因）

### 现象

- 问题二修复（按行拆分）之后，继续测试发现：连续发送 `1`、`2`、`3`、`4`（每次单独发送），OLED 和 Storage 里旧内容会不断重复出现、越滚越多，比如发送到第4条时屏幕上出现了 `4,3,2,1,3,2,1,...` 这种明显重复的序列。
- 清除历史记录（Setting 页面）之后，Storage 里很快又冒出很久以前测试用的旧数据（比如最早期测试的 `arch`），断电重启也没用。

### 排查过程

1. 对照串口调试助手打印出来的原始日志：第一次发送 `1,2,3`，回调报告 `Size=9`，内容正好是 `1\r\n2\r\n3\r\n`（3条×3字节）——这是对的。但紧接着只发送了一个 `4`，回调却报告 `Size=12`，内容是 `1\r\n2\r\n3\r\n4\r\n`——**上一次已经处理过的 `1,2,3` 又完整地出现了一遍**，`Size` 明显不是"这次新增了多少字节"，而是"从最初到现在的总字节数"。
2. 顺着这个线索去看 USART1 的 RX DMA 配置：`hdma_usart1_rx.Init.Mode = DMA_CIRCULAR`（循环模式），而代码在每次 `HAL_UARTEx_RxEventCallback` 回调的最后，都会重新调用一次 `HAL_UARTEx_ReceiveToIdle_DMA(huart, uart_rx_buffer, UART_RX_BUFFER_SIZE)` 想"重新武装，准备接收下一条"。
3. 这是两种互相矛盾的用法混在一起了：`DMA_NORMAL`（非循环）模式下，每次重新调用这个函数确实会让 DMA 从缓冲区开头重新开始，`Size` 才会正确地只代表"这次新增的字节数"；但在 `DMA_CIRCULAR` 模式下，DMA 硬件本来就会不停地转下去，重新调用这个函数只会让 HAL 库内部记录的"期望长度"跟硬件实际已经写到的位置对不上，`Size` 因此变成一个不断增长的"绝对位置"，而不是"增量"。
4. 这个问题从项目一开始配置 USART1 DMA 为循环模式起就存在，只是一直没暴露：改行拆分之前，代码把整块数据当成一条记录处理，`Size` 变大只会让记录变长变乱，不容易被注意到；改成按行拆分之后，重复的内容以"看得懂的独立行"的形式反复出现，问题才变得非常明显。
5. 顺带解释了"清除历史后又冒出旧数据"：既然每次串口事件都会把从最开始到现在的所有内容重新当成"新数据"处理一遍，那么整个调试过程里发过的每一行（包括很早期测试的 `arch`）都在被反复重新识别、反复调用"保存历史记录"，`StorageCmdQueue` 里长期堆积着海量重复的保存请求。清除操作本身是生效的，但清除完之后，排在后面的这些旧的重复保存请求接着执行，把旧内容又写了回去。

### 根因（结论）

`DMA_CIRCULAR`（循环）模式下，DMA 硬件的写入位置是持续递增（绕回缓冲区末尾后从0继续）的，`HAL_UARTEx_RxEventCallback` 的 `Size` 参数反映的是这个**绝对位置**，而不是"这次新增了多少"。如果沿用"非循环模式"下"每次回调都重新调用 `HAL_UARTEx_ReceiveToIdle_DMA` 从头开始收"的写法，会让 HAL 内部记录的起点和硬件实际的循环位置逐渐脱节，导致早就处理过的旧数据被当成新数据一遍遍重新处理。

### 修复

在 `Core/Src/usart.c` 里：
- 不再在回调里重新调用 `HAL_UARTEx_ReceiveToIdle_DMA`（只在 `UART_Rx_Start()` 里调用一次，配合 `DMA_CIRCULAR` 硬件会自己不停地收）。
- 自己维护一个 `uart_rd_pos`，记录"上次处理到循环缓冲区里的哪个绝对位置"。每次回调只处理 `uart_rd_pos` 到这次 `Size` 之间**真正新增**的字节；如果 `Size` 比 `uart_rd_pos` 还小，说明缓冲区绕回过一次开头，分两段（`[uart_rd_pos, 缓冲区末尾)` 和 `[0, Size)`）依次处理。
- 用一个跨事件持续存在的小缓冲区 `uart_line_acc` 拼接一行内容，逐字节喂入、遇到 `\n` 才算一行结束，这样即使一行内容恰好跨越了绕回边界，也能正确拼起来，不会丢字符。

### 经验教训

- **循环 DMA + 每次回调重新"武装"，是两种互不兼容的用法混用**：循环模式下 DMA 硬件从不停止，正确做法是自己记录上次处理到的位置，只处理增量；如果想用"每次回调重新开始收"这种更简单的写法，DMA 必须配置成非循环（Normal）模式。混用会导致 `Size` 语义从"增量"变成"绝对位置"，而代码却还在按"增量"的假设处理数据。
- 这类 bug 有很强的隐蔽性：只有当**下游逻辑真的会对"重复内容"敏感**（这里是按行拆分之后能认出"这是同一行内容"）时，才会暴露出来；下游逻辑越"糊涂"（比如把整块数据囫囵吞枣存成一条记录），bug 存在的时间越长都不会被发现。改进一处逻辑（按行拆分）反而"揭穿"了另一处更底层、更早就存在的 bug，这提示我们：修复问题后出现的新现象，不一定是新引入的 bug，也可能是原来就存在、只是被之前的实现掩盖住了。
- 排查"内容不断重复、越滚越多"这类问题时，要重点关注**某个位置/计数器是否该被重置的时候没有被重置**；这类 bug 的特征是"每次都会新增一点，但从不清空历史"，跟"完全不工作"或"用一次坏一次"的 bug 表现很不一样，容易被误以为是"偶尔的时序问题"而不是确定性的逻辑错误。
