# 项目学习与面试手册

这份手册用于按真实代码理解项目，并帮助在面试中清楚说明自己的设计、调试和 AI 协作方式。核心原则是：能指出代码位置、解释取舍，并区分“已验证”和“下一步计划”。

## 一、先建立全局图景

入口在 `Core/Src/main.c`：CubeMX 完成时钟和基础外设配置，随后启动 CMSIS-RTOS v2。`Core/Src/freertos.c` 创建 RTOS 对象与任务；`User/TASK/system_init.c` 中的高优先级 `InitTask` 完成依赖硬件的初始化。

初始化完成后，`System_Init_ReleaseTasks()` 设置 `SYSTEM_READY_FLAG`。每个业务任务先执行 `System_Init_WaitReady()`，并以 `osFlagsNoClear` 等待事件，因此任意一个任务不会把“系统已就绪”状态消费掉。`InitTask` 调用 `osThreadExit()` 后删除自身，正常任务才持续运行。

任务职责如下：

| 任务/模块 | 职责 | 关键接口 |
| --- | --- | --- |
| KeyScanTask | 扫描按键、产生 UI 输入 | `key_task.c` |
| UIManagerTask | 页面切换与 OLED 刷新 | `ui_task.c` |
| MPUReadTask | 读取 DMP 姿态数据 | `mpu_task.c` |
| StorageTask | 串口历史与元数据写入 Flash | `storage_task.c` |
| LedTask | 周期调用 WS2812 状态机 | `led_task.c`、`ws2812.c` |
| WatchdogTask | 统计各任务 check-in 决定是否喂狗 | `watchdog_task.c` |
| CanTask | 发送回环测试帧、消费接收队列 | `can_task.c` |

## 二、关键设计如何解释

### 1. InitTask 与同步门槛

目的不是“高优先级任务退出后其他任务自然能跑”这么简单，而是明确限制其他任务不能在 Flash、DMP 等依赖尚未可用时访问它们。事件标志被视为一个持续状态，而不是一次性通知，所以等待时必须使用 `osFlagsNoClear`。

面试表达：我把硬件初始化集中到一次性高优先级任务；业务任务通过事件标志等待依赖就绪。这样启动顺序可控，初始化失败时也能在 OLED/串口显示具体阶段。

### 2. WS2812：PWM + DMA

文件：`User/WS2812/ws2812.c`、`Core/Src/tim.c`。WS2812 用高低电平宽度编码 0/1，TIM3 产生固定周期 PWM，DMA 连续把每一位对应的 CCR 占空比写入定时器，无需 CPU 逐位延时。`WS2812_Tick()` 将静态、呼吸和流水等效果组织成非阻塞状态机；亮度、颜色、模式和开关被保存到 Flash 元数据并在上电后恢复。

面试表达：如果用普通延时循环驱动，任务调度和中断会破坏纳秒/微秒级时序；PWM 保证位周期，DMA 保证连续更新占空比，任务只负责准备缓冲区和触发传输。

### 3. Flash 存储

文件：`User/NORFLASH/norflash.c`、`User/TASK/storage_task.c`。存储任务将耗时的擦写从 UI/串口路径移出，避免交互任务被阻塞；历史与状态分开处理，上电时由 `Meta_Load()` 恢复。

面试表达：Flash 写入有擦除粒度和耗时，不能放在高频 UI 或中断上下文；我用任务和队列串行化写操作，避免并发访问与界面卡顿。

### 4. 看门狗不是“定时喂一下”

文件：`User/TASK/watchdog_task.c`。每个业务任务定期 `Watchdog_Checkin()`；看门狗任务每 500 ms 检查各任务最近一次活动时间，超过 1500 ms 就打印 stale 信息且不刷新 IWDG。这样只能证明“系统整体仍在推进”，而不是某一个看门狗任务还活着。

### 5. CAN 内部回环

文件：`User/TASK/can_task.c`。CanTask 每 500 ms 用 ID `0x321` 发一帧；`HAL_CAN_RxFifo0MsgPendingCallback()` 在接收中断中取 FIFO，并用 `osMessageQueuePut(..., 0)` 将帧交给任务；任务消费队列并更新 TX/RX/ERR/DROP 统计。

`CAN_MODE_LOOPBACK` 是控制器内部自检，不经过 CAN 收发器和 CANH/CANL。它验证的是 CAN 外设、过滤器、中断和 ISR-to-task 消息传递，不能回答“真实总线能否通信”。

## 三、真实踩坑与排查方法

### FreeRTOS 堆不足导致任务静默创建失败

现象是 StorageTask 不运行但系统不崩溃。排查时在任务首行打点，并检查 `osThreadNew()` 返回句柄和 `xPortGetFreeHeapSize()`；根因是 `configTOTAL_HEAP_SIZE` 不够。教训：链接器显示的 RAM 占用不等于 FreeRTOS 动态堆余量。

### `printf` 撑爆小任务栈

心跳任务原本栈很小，加入复杂格式化输出后可能触发栈溢出钩子，表现为全系统停摆。教训：诊断代码也消耗栈；应结合 `uxTaskGetStackHighWaterMark()` 调整栈，而不是仅凭经验。

### UART DMA Circular 与重复“重启接收”混用

循环 DMA 的回调位置是环形缓冲区中的绝对位置，不能按 Normal DMA 的“本次增量长度”理解。需要自己维护读指针，只处理新增区间，并处理回绕。

### 初始化后仍被 IWDG 周期复位

本项目曾把“系统就绪”事件当作锁存状态，却使用默认自动清除等待。第一个任务通过后清除了标志，其他任务（包括 WatchdogTask）永远阻塞，最终无人喂狗。修复是 `osFlagsNoClear`。教训：同步原语的“是否消费信号”是设计语义，不是小参数。

更完整的历史排查过程见 `docs/debugging-notes/`。

## 四、CubeMX 与代码维护边界

CubeMX 可重生成文件，因此自定义代码放入对应的 `USER CODE BEGIN/END` 区域。任务参数、外设引脚、DMA 模式等应同步回 `.ioc`，而非只改生成的 `.c` 文件。手动创建的 CanTask 位于 `freertos.c` 的用户代码区；这是有意的最小侵入方案，后续也可迁移为 CubeMX 管理的任务。

## 五、AI 协作实践：如何诚实地讲

项目早期保留了 Claude Code 阶段的调试记录；当前迭代中使用 Codex 协助阅读工程、提出初始化/CAN 方案、生成局部代码、执行构建检查和整理文档。AI 的价值是缩短资料检索、代码导航与方案对比时间，不是替代硬件验证或工程决策。

我自己的工作仍包括：描述真实现象与约束、在 CubeMX 中完成引脚/DMA/RTOS 配置、烧录到板上验证、判断日志是否符合预期、决定是否接受改动，并把失败过程保留下来。例如 CAN 先用内部回环验证软件链路，不能因为 TX/RX 计数增长就声称完成真实总线通信。

可在面试中这样说：

> 我把 Agent 当作结对工程助手，而不是答案生成器。我会先给它项目约束，例如 CubeMX 用户代码边界和硬件实测结果；让它协助定位可疑调用链、提出最小改动和验证命令。每次修改后仍由我在开发板上复现与验收。对 AI 给出的结论，我会检查源码、构建输出和硬件日志；没有实测证据的内容会明确标为待验证。

### Prompt、Skill 与 Subagent 的心得

- Prompt 应包含目标、现象、已有日志、不能改动的边界和验收标准。只说“修一下”会让方案不可控。
- Skill/流程的作用是强制执行设计、计划、调试、验证和提交前检查；它们适合降低遗漏，不代替对 MCU 时序、堆栈和硬件的理解。
- Subagent 适合拆分相互独立的查阅、文档或测试任务；共享同一工作区时必须明确文件所有权，避免并发修改同一文件。本次文档任务规模小，采用单会话执行并通过独立 worktree 隔离。
- 最有价值的产物不是一次成功的回答，而是可复现的日志、最小修复、构建结果和调试复盘。

## 六、面试速答清单

1. **为什么要 RTOS？** 多外设工作频率和阻塞特性不同，用任务隔离职责；队列、事件和互斥机制表达数据与依赖关系。
2. **中断里为什么不直接更新 OLED？** OLED 刷新慢且可能阻塞；中断只取关键数据并入队，任务在可控上下文消费。
3. **DMA 的收益？** 降低 CPU 持续搬运数据的负担，并使 WS2812 时序或 UART 接收更稳定；仍要处理缓冲区边界和并发语义。
4. **看门狗如何避免误喂？** 检查多个任务的活跃时间，任一关键任务超时就停止喂硬件狗。
5. **你还会如何完善？** 用第二节点验证真实 CAN，增加错误恢复、栈水位和队列监控，并补齐硬件接线图与自动化硬件测试。
